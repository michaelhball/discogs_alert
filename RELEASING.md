# Releasing

How to cut a new release of `discogs_alert` to PyPI and DockerHub.

## Quick recipe

1. **Open a "Bump version" PR** that updates `version` in `pyproject.toml` to
   the new `X.Y.Z`. Title: `Bump version to X.Y.Z`. Merge it (CI green).
2. **Tag the merge commit** with `vX.Y.Z` and push the tag:
   ```bash
   git checkout main && git pull
   git tag -a vX.Y.Z -m "Release X.Y.Z"
   git push origin vX.Y.Z
   ```
3. The `release.yml` GitHub Actions workflow runs on the tag push and:
   - builds the wheel + sdist with Poetry,
   - uploads them to PyPI via OIDC trusted publishing (no token needed),
   - drafts a GitHub Release with autogenerated notes,
   - (optionally) builds and pushes the Docker image to
     `miggleball/discogs_alert:X.Y.Z` and `:latest`.
4. **Verify on PyPI:** https://pypi.org/project/discogs-alert/ should show the
   new version within a minute or two.
5. **Smoke-test on a fresh venv:**
   ```bash
   python -m venv /tmp/da-smoke && /tmp/da-smoke/bin/pip install discogs-alert==X.Y.Z
   /tmp/da-smoke/bin/python -m discogs_alert --help
   ```

## One-time setup (already done; here for reference)

### PyPI trusted publishing

1. Create the project on PyPI if it doesn't exist:
   https://pypi.org/manage/account/publishing/
2. Add a "Pending publisher" entry with:
   - PyPI project name: `discogs-alert`
   - Owner: `michaelhball`
   - Repository: `discogs_alert`
   - Workflow: `release.yml`
   - Environment: `pypi` (matches the workflow file)
3. Once the first tagged release runs, PyPI promotes the pending publisher to
   a real one. Future runs use OIDC tokens automatically — no API tokens to
   leak or rotate.

### DockerHub (optional)

The release workflow can push to DockerHub if the repository has the secrets
`DOCKERHUB_USERNAME` and `DOCKERHUB_TOKEN`. If those aren't set, the docker
job is skipped.

To enable:

1. Create an access token at https://hub.docker.com/settings/security with
   "Read & Write" scope.
2. In the repo settings → Secrets and variables → Actions, add:
   - `DOCKERHUB_USERNAME`: your DockerHub username (e.g. `miggleball`)
   - `DOCKERHUB_TOKEN`: the token you just created.

## Manual fallback (if the workflow can't run)

```bash
# Build
poetry install
poetry build

# Publish to PyPI (requires PYPI_TOKEN in your shell)
poetry publish --username __token__ --password "$PYPI_TOKEN"

# Build & push Docker
docker build -t miggleball/discogs_alert:X.Y.Z -f docker/Dockerfile .
docker tag miggleball/discogs_alert:X.Y.Z miggleball/discogs_alert:latest
docker push miggleball/discogs_alert:X.Y.Z
docker push miggleball/discogs_alert:latest
```

## Changelog / release notes

The release workflow uses GitHub's autogenerated release notes. To make those
useful:

- Land each PR with a one-sentence title that reads well in a changelog.
- For breaking changes, add a `## Breaking change` section in the PR body so
  the user sees it when they pull a release.

If you want a hand-curated changelog, summarise the merged PRs since the
previous tag:

```bash
git log $(git describe --tags --abbrev=0)..HEAD --oneline
```

## Versioning policy

`0.0.x` is the current trajectory. Bump to `0.1.0` when the next release
contains user-visible breaking changes (e.g. removed CLI flag, changed env
var, removed Python version). Bump to `1.0.0` when the project is ready to
commit to API stability.
